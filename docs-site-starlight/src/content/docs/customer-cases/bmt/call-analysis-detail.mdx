## 결정적 발견: Page 3은 순수 이미지

<Aside type="danger" title="pdfplumber 사용 불가 — OCR/비전 AI 필수">
GAR 배치도 3페이지를 `PyMuPDF`로 분석한 결과, **텍스트 레이어가 없는 순수 래스터 이미지**입니다.
pdfplumber로는 아무것도 추출할 수 없습니다. TAG 추출에 **OCR 또는 비전 AI가 반드시 필요**합니다.

이전 분석에서 "벡터 PDF라서 pdfplumber로 100% 추출"이라고 했지만, 그건 VD 상세도면(4~13페이지) 이야기였습니다.
고객이 핵심이라고 말한 3페이지는 **이미지**입니다.
</Aside>

| 페이지 | 형태 | 텍스트 레이어 | pdfplumber | 비고 |
|--------|------|-------------|-----------|------|
| 1~2 | 벡터 | ✅ 있음 | ✅ 가능 | GAR 표지/범례 |
| **3 (핵심)** | **래스터 이미지** | **❌ 없음** | **❌ 불가** | **GAR 배치도 — 고객이 "이거만 보면 된다"** |
| 4~13 | 벡터 | ✅ 있음 | ✅ 가능 | VD 상세도면 (이전 분석 대상, 핵심 아님) |
| 14~22 | 벡터 | ✅ 있음 | ✅ 가능 | ITDRA 계장기기도 |

### 📞 "코드만 일치하는지 여부만 보면 되는 거예요. 인치나 이런 게 필요한 게 아니고"

**실제 확인**: 이것이 의미하는 바:

| 비교 방식 | 필요? | 근거 |
|----------|------|------|
| TAG → 품목코드 → BOM 존재 여부 | ✅ **핵심** | "구성품들이 제대로 BOM에 들어갔냐 안 들어갔냐" |
| 재질 비교 (A182-F316L 등) | ❌ 불필요 | "코드가 그런 걸 다 담고 있거든요" |
| 치수 비교 (2170mm 등) | ❌ 불필요 | "몸통에서의 인치는 전혀 필요가 없고" |
| 개별 부품 비교 (Body, O-Ring) | ❌ 불필요 | 배치도 TAG 단위가 핵심 |

<Aside type="note" title="치수에 대한 뉘앙스">
박순근 팀장님: "원래는 치수도 보면 좋아요. 8인치 쓸 거를 6인치라고 넣을 수도 있고. 그런데 그것까지 될지는 의문이긴 해요."

→ 치수 검증은 **nice-to-have**이지 필수가 아닙니다. 1차 과제 범위에서는 제외하되, 향후 확장 가능성은 열어둡니다.
</Aside>

**의미**: 현재 22페이지 합본 PDF는 샘플용이고, 실제 운영 시에는:

### 📞 "AI 툴이 알아서 프론트 뷰에 있는 거는 이 에리어에 이게 프론트 뷰구나 하고 캐치하면 되는 거 아니에요"

**실제 확인**: GAR 배치도에는 여러 뷰가 있습니다:
- Top View (위에서 본 모습)
- Front View (앞에서 본 모습)
- A View / Right View / Bottom View

각 뷰에 TAG가 흩어져 있고, **뷰 레이아웃이 매번 고정되지 않습니다** (팀장님: "맵으로 잡으면 안 돼요. 매번 딱 저기 탑뷰가 있는 게 아니라").

이것이 **AI R&D가 필요한 핵심 이유**입니다:
1. 이미지에서 TAG 텍스트 인식 (OCR)
2. 뷰 영역 자동 분류 (레이아웃 분석)
3. TAG와 치수 구분 (알파벳 시작 = TAG, 숫자만 = 치수)

## 발견된 실제 불일치 (자동 검출 가능)

Part List ↔ ERP BOM 간 실제 불일치가 **이미 존재**합니다. 이것이 고객이 찾고 싶어하는 것입니다.

| # | TAG | Part List 코드 | ERP BOM 코드 | 불일치 유형 |
|---|-----|---------------|-------------|-----------|
| 1 | **V06** | `GPCV-RDN50PN16-VARIBELL` | `GPCV-RDN50PN16-**DNV**-VARIBELL` | 선급코드(`-DNV`) 누락 |
| 2 | **PI02** | `PI-DH-MAX**331**` | `PI-DH-MAX**311**` | 모델번호 오타 (331↔311) |
| 3 | **Double Wall Flange** | `*-**34L**-DNV` | `*-**36L**-DNV` | 길이 코드 불일치 (34L↔36L) |
| 4 | **Limit Switch** | VAA-L-* (조달코드) | LS-F25-* (Maker 코드) | 코드 체계 완전 상이 |

<Aside type="tip" title="이것이 고객이 실제로 찾고 싶은 것">
위 4건 같은 **코드 불일치**가 바로 "BOM에 등록된 게 누락된 게 있나 없나를 찾는" 과정에서 발견되는 것입니다.
V06의 `-DNV` 누락이나 PI02의 331↔311 같은 건 **실제 에러일 가능성**이 높습니다.
</Aside>

## 기술 요구사항 정리

| 단계 | 기술 | 난이도 | R&D 여부 |
|------|------|--------|---------|
| TAG 추출 (GAR 이미지) | OCR + 좌표 기반 텍스트 인식 | **높음** — 래스터 이미지, 다중 뷰, 회전 텍스트 | ✅ **R&D 필요** |
| TAG vs 치수 구분 | 규칙 기반 (알파벳 시작 = TAG) | 낮음 | ❌ |
| Part List 파싱 | zipfile + xml (기존 파서) | 낮음 | ❌ |
| 코드 매칭 | exact match + fuzzy | 낮음 | ❌ |
| 뷰 영역 자동 분류 | 이미지 레이아웃 분석 | **높음** — 매번 레이아웃 다름 | ✅ **R&D 필요** |
| 누락 리포트 생성 | openpyxl (기존 리포터) | 낮음 | ❌ |
