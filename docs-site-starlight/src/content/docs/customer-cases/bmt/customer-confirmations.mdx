## 수령 자료 & 용어 정리

아래 확인 사항을 읽기 전에 필요한 배경 지식입니다.

### 수령 파일 구조

BMT에서 총 3개 그룹, 7개 파일을 수령했습니다.

| 그룹 | 파일 | 설명 |
|------|------|------|
| **1번 — 단품 밸브** | `FB2FB6-...-36L-KGS.pdf` (1페이지) | 밸브 1개의 조립도면 + 부품리스트 |
| | `FB2FB6-...__BOM.xlsx` | ERP에 등록된 이 밸브의 BOM |
| | `FB2FB6-..._BOM&파트리스트.xlsx` | 고객이 수동으로 불일치 7건 표시한 참고 파일 |
| **2번 — 스키드(GVU)** | `W5XGVU-SN2709-GARRB.pdf` (22페이지) | 밸브 8개 + 센서 + 배관을 합친 시스템 도면 |
| | `W5XGVU-SN2709-PLRA.xlsx` (4시트) | 밸브/센서/피팅/기술사양 Part List |
| | `W5XHEGVU-SN2709-DNV.xlsx` | ERP에 등록된 이 스키드의 BOM |
| **3번 — 참조** | `GVU_3D.pdf` | 3D 렌더링 (TAG 라벨 확인용, 나머지 파일은 2번과 동일) |

### 파일명 읽는 법

**1번 그룹 — `FB2FB6-32A-1S-D011-1SDNA-36L-KGS`**

| 파일명 요소 | 의미 |
|------------|------|
| `FB2FB6` | 제품 모델명 (Ball Valve 시리즈) |
| `32A` | 밸브 사이즈 (32A = 1¼인치) |
| `36L` | 밸브 길이 |
| `KGS` | 한국가스안전공사 인증 |

밸브 1개짜리 도면이므로 **"프로세스밸브 (단품)"** 구분에 해당합니다.

**2번 그룹 — `W5XGVU-SN2709-GARRB`**

| 파일명 요소 | 의미 |
|------------|------|
| `W5X` | 프로젝트/선종 코드 |
| `GVU` | **Gas Valve Unit** — 밸브 여러 개를 프레임에 조립한 시스템 |
| `SN2709` | **Serial Number 2709** — 선박 1척의 고유 ID (호선번호) |
| `GARRB` | GAR(General Arrangement, 배치도) + RB(Revision B) |

GVU는 밸브 여러 개(V01~V15)를 포함하는 시스템이므로 **"스키드 (시스템)"** 구분에 해당합니다.

### 22페이지 PDF 내부 구조

2번 그룹의 도면 PDF(22페이지)는 3개 섹션이 합본되어 있습니다.

| 섹션 | 약어 | 정식 명칭 | 페이지 | 내용 |
|------|------|----------|--------|------|
| 배치도 | **GAR** | General Arrangement | 1~3 | 스키드 전체 배치, 밸브 위치 표시 |
| 밸브 상세도 | **VD** | Valve Drawing | 4~13 | 개별 밸브 8개의 단면도 + 부품리스트 |
| 계장기기도 | **ITDRA** | Instrument/Tubing Drawing | 14~22 | 온도센서, 압력센서 등 계장기기 스펙 |

### SN(호선번호)이란?

**SN = Serial Number = 호선번호**로, 조선소에서 건조하는 선박 1척의 고유 ID입니다.

- `SN2709`와 `SN2693`은 **서로 다른 배**입니다
- 같은 GVU 모델이라도 호선마다 사양이 다를 수 있습니다
- 따라서 SN2693 도면의 부품을 SN2709 BOM과 비교하면 "진짜 불일치"와 "호선 차이"를 구분할 수 없습니다

이것이 아래 블로커 1번(VD 도면 SN 불일치)이 치명적인 이유입니다.

### SN2709용 VD 도면 재전달

| 항목 | 내용 |
|------|------|
| 상태 | 🔴 요청 완료, 미수령 |
| 출처 | 📞 통화 + 🔍 내부 분석 |
| 요청일 | 2026-03-24 |
| 영향 | GVU 스키드의 VD↔ERP BOM 비교가 **완전 차단** |

**발견 경위**: POC 테스트 중 PDF 내부 DOC 번호를 `PyMuPDF`로 추출했더니, VD 섹션만 다른 호선 번호가 나왔습니다.

GAR 표지(Page 1)에는 **SN2709**:

![GAR 표지 — SN2709 정상](/images/bmt/gvu-cover-sn2709.png)

VD 표지(Page 4)에는 **SN2693** — 다른 호선:

![VD 표지 — SN2693 불일치](/images/bmt/vd-cover-sn2693.png)

| 섹션 | 페이지 | DOC 번호 내 SN | 상태 |
|------|--------|---------------|------|
| GAR | 1~3 | **SN2709** | ✅ 정상 |
| **VD** | **4~13** | **SN2693** | **❌ 불일치** |
| ITDRA | 14~22 | **SN2709** | ✅ 정상 |
| Part List | 엑셀 | **SN2709** | ✅ 정상 |
| ERP BOM | 엑셀 | **SN2709** | ✅ 정상 |

📞 **통화 확인 내용**: "파일명은 SN2709인데 VD 내부 DOC No.만 SN2693인 것은 잘못 보낸 것으로 추정" — 재전달 요청 완료.

**현재 상태**: VD 테이블 추출(151개 부품)과 Balloon 교차검증은 완료했으나, SN이 다르면 BOM 비교 결과를 신뢰할 수 없어 **파이프라인이 자동 스킵** 중입니다.

**요청 사항**: `W5XGVU-SN2709-VD` 도면을 전달해 주시면 즉시 BOM 비교를 실행할 수 있습니다.

<Aside type="tip" title="실제 영향 범위 — 전체 판단에는 문제 없음">
스키드(GVU) 검증은 3단계로 나뉘는데, VD 불일치로 막힌 부분은 **2단계뿐**입니다.

| 단계 | 내용 | SN 상태 | 테스트 가능 여부 |
|------|------|---------|----------------|
| **1단계** | 도면 TAG(V01) → Part List CODE → ERP BOM 자품목코드 | GAR/Part List/ERP 모두 **SN2709** | ✅ 가능 |
| **2단계** | 개별 밸브 VD 테이블 ↔ ERP BOM 하위 부품 비교 | VD만 **SN2693** | ❌ 차단 |
| **3단계** | ITDRA 계장기기 스펙 ↔ Part List 센서 정보 | 둘 다 **SN2709** | ✅ 가능 |

또한 2단계의 비교 로직(도면 부품리스트 ↔ BOM 코드 매칭)은 이미 **1번 자료 FB2FB6에서 검증 완료**된 것과 동일한 방식입니다. 호선만 맞으면 같은 코드가 그대로 동작합니다.

**결론**: VD 재수령은 "파이프라인이 동작하는지 증명"하기 위한 것이 아니라, **SN2709 호선의 실제 불일치를 찾아내는 실전 결과**를 얻기 위해 필요합니다.
</Aside>

### 도면 PDF가 항상 벡터(CAD 출력)인지?

| 항목 | 내용 |
|------|------|
| 상태 | 🔴 미확인 |
| 출처 | 🔍 내부 분석 — 수령 자료가 전부 벡터 PDF여서 의문 제기 |
| 영향 | **솔루션 아키텍처가 완전히 달라짐** |

**벡터 PDF란**: Inventor 같은 CAD에서 직접 출력한 PDF. 텍스트와 도형이 좌표 데이터로 저장되어 있어 확대해도 선명하고, 글자를 복사/검색할 수 있습니다.

현재 수령한 FB2FB6 도면은 벡터 PDF입니다. 텍스트 값 자체는 `pdfplumber`로 100% 정확하게 추출됩니다 (6개 도면 검증 완료). 단, 병합셀 구조 파싱에는 후처리가 필요합니다:

![FB2FB6 부품리스트 테이블 — pdfplumber 추출 대상](/images/bmt/fb2fb6-parts-table.png)

**이미지(스캔) PDF와의 차이**:

| 도면 형식 | 텍스트 추출 | 정확도 | 필요 기술 | 개발 범위 |
|-----------|-----------|--------|----------|----------|
| **벡터 PDF** (CAD 출력) | `pdfplumber` 직접 읽기 | **텍스트 100%** (구조는 후처리 필요) | 문자열 매칭 + 병합셀 split | ✅ 현재 완료 |
| **이미지 PDF** (스캔본) | OCR 필수 | 85~95% | eDOCr2 + 후처리 | ❌ 추가 개발 필요 |

**요청 사항**: 실무에서 도면을 다룰 때 아래 중 어떤 경우인지 확인 부탁드립니다.

1. **항상 CAD에서 PDF 출력** → 현재 방식으로 충분
2. **대부분 CAD이지만 가끔 종이 스캔** → OCR fallback 추가 필요
3. **스캔본이 많음** → OCR 기반으로 재설계 필요

### GA/Layout 도면 샘플 (부품번호 없는 유형)

| 항목 | 내용 |
|------|------|
| 상태 | 🟡 미수령 |
| 출처 | 📧 메일 회신 — "부품번호 없는 도면도 있다" 확인 |
| 영향 | Balloon도 Description 테이블도 없을 수 있어 **현재 파이프라인으로는 처리 불가** |

📧 **메일 회신 원문** (질문 2에 대한 답변): "부품번호가 표기되지 않은 도면 유형도 존재합니다 — GA / Layout 도면"

현재 파이프라인은 아래 두 가지를 기반으로 동작합니다:
1. Description 테이블 (NO./DESCRIPTION/MATERIAL/Q'TY)
2. Balloon 번호 (도면 이미지의 말꼬리표 숫자)

GA/Layout 도면에 이 둘이 없으면 별도 접근법이 필요합니다.

<Aside type="note" title="아래 'BOM에 있고 도면에 없는 정보 리스트(Q9)'와의 관계">
GA/Layout 도면이 실제로 어떤 용도인지는 샘플을 받아봐야 알 수 있습니다.

| 가능성 | 상황 | 우리 대응 |
|--------|------|----------|
| 대응 BOM이 있다 | 부품번호 없이 BOM만 존재 → "BOM에 있고 도면에 없는 정보"에 해당 | 별도 비교 로직 필요 |
| 대응 BOM이 없다 | 도면도 BOM도 없음 → 비교 대상 자체가 아님 | 파이프라인에서 제외 |
| 참조용 도면이다 | BOM 검증과 무관한 레이아웃 참고용 | 파이프라인에서 제외 |

Q5(도면 샘플)를 수령하면 어떤 경우인지 판단할 수 있고, Q9(엑셀 리스트)를 수령하면 GA/Layout 관련 항목이 포함되어 있는지 교차 확인할 수 있습니다.
</Aside>

**요청 사항**: GA/Layout 도면 샘플 1개를 전달해 주시면, 이 유형의 구조를 파악하고 대응 방안을 설계하겠습니다.

### Part List 존재 형태

| 항목 | 내용 |
|------|------|
| 상태 | 🟡 미확인 |
| 출처 | 📧 메일 회신 — 질문 5에 대한 보정 답변에서 발생 |
| 영향 | Part List가 별도 엑셀 파일이면 3자 비교 구조가 필요 |

📧 **메일 회신 원문** (질문 5 보정): "도면 Balloon No → **Part List (부품번호)** → ERP BOM 코드" — 중간에 Part List가 별도로 존재한다는 의미.

현재 1번 자료(FB2FB6)에서는 도면 PDF 안의 부품리스트가 곧 Part List였습니다. 하지만 고객 보정에 따르면 별도 Part List가 있을 수 있습니다.

| 구조 | 비교 방식 |
|------|---------|
| 도면 PDF 안에 Part List 포함 | 도면 ↔ ERP BOM (2자 비교) |
| Part List가 별도 엑셀 | 도면 ↔ Part List ↔ ERP BOM (3자 비교) |

**요청 사항**: 프로세스밸브(단품)의 경우에도 별도 Part List 엑셀이 존재하는지 확인 부탁드립니다.

Part List 엑셀(`W5XGVU-SN2709-PLRA`) 내부에 `W5XGVU-SN2711-PLRA` 참조가 발견되었습니다.

| 자료 | SN |
|------|-----|
| 도면 PDF 파일명 | SN2709 |
| Part List 파일명 | SN2709 |
| **Part List 내부 참조** | **SN2711** |
| ERP BOM | SN2709 |

**요청 사항**: SN2711은 SN2709와 같은 사양(동일 BOM)인지, 아니면 다른 호선(다른 BOM)인지 확인 부탁드립니다. 혹시 SN2709/SN2710이 자매호선이고 SN2711은 다른 프로젝트인지도 확인이 필요합니다.

### BOM에 있고 도면에 없는 정보 리스트

| 항목 | 내용 |
|------|------|
| 출처 | 📞 통화 — "BOM에 있고 도면에 없는 정보를 엑셀로 리스트업 하겠다" |
| 용도 | 시스템이 자동 검출한 `bom_only` 항목과 대조하여 정확도 검증 |

현재 파이프라인은 이미 `bom_only` (BOM에만 존재) 항목을 자동 검출합니다. 고객이 수동으로 정리한 리스트와 대조하면 **false positive/negative를 측정**할 수 있습니다.

<Aside type="note" title="위 'GA/Layout 도면 샘플(Q5)'과의 관계">
이 리스트의 항목이 어디서 발생하는지에 따라 대응이 달라집니다.
- **일반 도면(FB2FB6, VD)에서 빠진 항목** → 단면도에 가려진 부품 등 (현재 `bom_only`로 자동 검출 중)
- **GA/Layout 도면 관련 항목** → GA/Layout에 대응 BOM이 있는지 자체가 불명확 (Q5 샘플로 확인 필요)

이 리스트와 Q5 샘플을 함께 수령하면 전체 그림을 파악할 수 있습니다.
</Aside>

### 🟠 내부 미해결 — 재질 비교 미구현

| # | 질문 (내부 + 고객) | 현재 상태 | 왜 중요한가 |
|---|-------------------|----------|------------|
| Q11 | 재질/규격 비교 파서 확장이 필요 | ⚠️ 파서 수정 필요 (고객 질문 아님) | ERP BOM에 **E열(규격), F열(재질)**이 존재하는 것을 확인. 현재 파서가 A/C/D/L만 읽고 E/F를 무시 중. 파서에 E/F열을 추가하면 도면 MATERIAL ↔ ERP 재질 직접 비교 가능 |

<Aside type="caution" title="재질 비교 — ERP에 F열(재질) 존재 확인, 파서 미구현">
**ERP BOM 엑셀을 다시 확인한 결과, E열(규격)과 F열(재질)이 존재합니다.** 현재 파서가 이 컬럼을 무시하고 있었습니다.

**ERP BOM 엑셀 전체 컬럼 구조 (A~AD, 30개 컬럼):**

| 컬럼 | 헤더 | 예시 값 | 현재 파서 |
|------|------|---------|----------|
| A | 단계 | `.1` | ✅ 사용 중 |
| B | 순번 | `1` | ❌ 무시 |
| C | 자품목 | `BY-FB2F26-32A-1S-36L-KGS` | ✅ 사용 중 |
| D | 자품목명 | `BODY / KGS` | ✅ 사용 중 |
| **E** | **규격** | `2 CL150 BW SCH10S` | **❌ 무시 — 추가 필요** |
| **F** | **재질** | `A182-F316L` | **❌ 무시 — 추가 필요** |
| L | 자품목기준수 | `1` | ✅ 사용 중 (수량) |
| Y | 비고 | `V01` (TAG 참조) | ❌ 무시 — TAG 매핑에 활용 가능 |

**즉, 재질 직접 비교가 가능합니다:**

| 도면 MATERIAL | ERP F열(재질) | 비교 가능? |
|--------------|-------------|----------|
| A182-F316L | A182-F316L | ✅ exact match 가능 |
| VITON | (확인 필요) | 부품코드 접미사 V와 F열 값 교차 확인 |
| SS316 | (확인 필요) | 부품코드 접미사 8M과 F열 값 교차 확인 |

**필요 조치**: 파서에 E열(규격)과 F열(재질)을 추가하면 도면 MATERIAL ↔ ERP 재질 직접 비교가 가능합니다.
추가로 Y열(비고)에 TAG 번호(`V01`)가 들어있어 VALVE LIST 매칭에도 활용할 수 있습니다.
</Aside>

## 기술 현실 분석 — R&D가 필요한가?

### 현재 파이프라인에 사용되는 기술

| 기술 | 사용 여부 | 실제 역할 |
|------|----------|----------|
| LLM (GPT, Claude 등) | ❌ 사용 안 함 | - |
| YOLO (객체검출) | ❌ 사용 안 함 | 밸브 단면도에서 부품 분리 불가 → 통화 합의로 제외 |
| OCR | ❌ 사용 안 함 | 벡터 PDF라 필요 없음 |
| pdfplumber | ✅ | PDF 텍스트 레이어에서 테이블 추출 (텍스트 복사 수준) |
| PyMuPDF | ✅ | SN 추출, Balloon 번호 보조 |
| 문자열 매칭 | ✅ | exact + prefix + Levenshtein + 동의어 사전 |
| 정규식 | ✅ | SN 추출, 파일명 파싱 |
| zipfile + xml | ✅ | 엑셀 파싱 |

<Aside type="caution" title="솔직한 현실">
현재 BMT 파이프라인은 **시스템 이해 + Python 파싱 + 문자열 비교**입니다.
AI/ML 모델을 사용하지 않으며, 벡터 PDF + 정형 데이터 구조 덕분에 기존 라이브러리만으로 동작합니다.
경쟁 업체가 "OCR로 읽는 거 어렵지 않다"고 한 것도 이 맥락에서는 맞는 말입니다 — 벡터 PDF에선 OCR조차 불필요합니다.
</Aside>

### R&D가 필요한 조건 — 고객 답변에 따라 결정됨

아래 표에서 **Q2(벡터/스캔 여부)**와 **Q5(GA/Layout 샘플)** 답변이 R&D 범위를 결정합니다.

| 고객 답변 시나리오 | R&D 필요 여부 | 근거 |
|-------------------|-------------|------|
| Q2 = "항상 CAD 출력" + Q5 = "참조용 도면" | **R&D 불필요** | pdfplumber + 문자열 비교로 전부 해결. 시스템 엔지니어링 프로젝트 |
| Q2 = "가끔 스캔본 있음" | **OCR R&D 필요** | eDOCr2/PaddleOCR + 후처리 파이프라인 개발 |
| Q5 = "BOM이 있는 도면인데 부품번호가 없음" | **도면 이미지 분석 R&D 필요** | 비정형 도면에서 정보 추출 → AI 모델 투입 가능성 |
| 다른 고객 확장 시 | **범용화 R&D 필요** | 고객마다 도면 포맷, 코드 체계, BOM 구조가 다름 |

### BMT 단독으로 볼 때 R&D 가능성이 있는 영역

| 영역 | 현재 상태 | R&D 요소 | 실현 조건 |
|------|----------|---------|----------|
| **스캔본 PDF OCR** | 미확인 (Q2 답변 대기) | OCR 엔진 + 도면 특화 후처리 | BMT에서 "스캔본도 있다"고 답변 시 |
| **ITDRA 구조화 파싱** | 현재 실패 중 | 비정형 테이블 인식 | 템플릿 기반으로 풀릴 수 있어 R&D라 하기 애매 |
| **동의어 사전 자동 학습** | 수동 1건 (ANSI↔HB) | 매칭 데이터 축적 → 자동 매핑 생성 | 데이터가 수십~수백 건 쌓여야 의미 있음 |
| **불일치 원인 자동 분류** | 미구현 | "재질 오류" vs "명명 규칙 차이" 자동 판별 | 에러 유형 데이터(Q8) 수령 후 |
| **GA/Layout 도면 처리** | 미확인 (Q5 샘플 대기) | 부품번호 없는 도면에서 정보 추출 | 도면 이미지 분석이 필요할 경우에만 |

### BMT를 넘어 확장할 때 R&D 가치가 생기는 영역

| 영역 | 설명 | 왜 R&D인가 |
|------|------|-----------|
| **다양한 도면 포맷 범용 파서** | 고객마다 CAD 소프트웨어, 출력 설정, 테이블 구조가 다름 | 매번 수동으로 파서를 만들 수 없으므로 자동 테이블 인식이 필요 |
| **코드 체계 자동 매핑** | 고객마다 사내 부품코드 규칙이 다름 (BMT: FB2FB6, 다른 고객: 완전히 다른 체계) | 동의어 사전을 수동으로 만들 수 없으므로 자동 학습 필요 |
| **스캔본/손글씨 도면 OCR** | BMT는 벡터지만 다른 고객은 스캔본일 수 있음 | 도면 특화 OCR은 기존 범용 OCR과 다른 전처리/후처리 필요 |
| **검증 결과 신뢰도 판정** | "이 불일치가 실제 에러인가, 명명 규칙 차이인가" 자동 판별 | 도메인 지식 + 패턴 학습 필요 |

### 결론

<Aside type="danger" title="핵심 판단">
**BMT 단독 건**은 고객 답변(Q2, Q5)이 "벡터 PDF만 / GA는 참조용"이면 **R&D 요소가 거의 없습니다.**
시스템 이해도 + Python 파싱 능력이 있으면 구현 가능한 **시스템 엔지니어링 프로젝트**입니다.

**R&D 가치는 다음 두 가지에서 발생합니다:**
1. **BMT 내부**: Q2="스캔본 있음" 또는 Q5="BOM 있는데 부품번호 없는 도면" → AI/OCR 투입 필요
2. **BMT 외부**: 다른 고객으로 확장 시 도면 포맷/코드 체계가 다양해져 범용화 R&D 필요

따라서 **AI 바우처 사업계획서**를 작성할 때, BMT 1개 고객만으로는 AI R&D 정당화가 어려울 수 있으며,
**다양한 고객/도면 포맷에 대응하는 범용 검증 시스템**으로 범위를 잡아야 R&D 명분이 생깁니다.
</Aside>
