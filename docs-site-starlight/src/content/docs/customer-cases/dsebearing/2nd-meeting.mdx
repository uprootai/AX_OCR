</Aside>

## 핵심 발견: 추출 범위 대폭 축소

### 고객 확인 — "외경·내경·폭, 딱 3개만"

도면을 함께 보면서 어떤 치수가 필요한지 하나씩 확인했습니다.

<Aside type="tip" title="고객 원문">
*"세부 숫자들은 관계가 없고 결국 외경과 내경 딱 2개 하고 폭까지 그냥 세 가지만 딱 뽑아내서 리스트화시키는 것이 가장 최종 목적이다."*
</Aside>

| 영역 | 기존 이해 | 2차 미팅 확정 |
|------|----------|-------------|
| **추출 대상** | 도면의 모든 치수 (548개) | **외경·내경·폭 3개만** |
| **세부 치수** (C1, 20, 160 등) | 전부 추출 | ❌ 관계 없음 |
| **상단 공차 테이블** | 불필요 (1차 확인) | ❌ 불필요 재확인 |
| **NOTES / ISO VIEW** | 불필요 (1차 확인) | ❌ 불필요 재확인 |
| **가공 표기** | 나중에 | 2단계 |

### 도면별 OD/ID/폭 확인

미팅에서 2개 도면을 함께 보며 구체적으로 어떤 값이 OD/ID/폭인지 확인했습니다.

| 도면 | 외경(OD) | 내경(ID) | 폭(W) | 비고 |
|------|---------|---------|--------|------|
| TD0062017 (BEARING RING T1) | ~650 | 확인 필요 | 확인 필요 | 화면이 작아서 정확한 값은 별도 확인 |
| 다른 도면 1종 | 확인 | 확인 | 확인 | *"다른 것도 사실상 다 이런 포맷"* |

### 기존 시스템 시연 — 전체 추출 방식 설명

기존 시스템의 전체 치수 추출 → 검증 UI 흐름을 시연했습니다.

> *"원래 컨셉은 모든 숫자를 다 읽어서 추출한 것들을 신뢰도 기반으로 자동 승인하고, 나머지만 사람이 보는 방식이었는데, 말씀 듣고 보니 순차적으로 먼저 외경 내경 폭부터 먼저 구하는 게 가장 급선무 같다."*

| 시연 내용 | 고객 반응 |
|----------|----------|
| 도면 위 OCR 오버레이 (전체 치수) | 확인 |
| 틀린 값 수정 → 승인 흐름 | 확인 |
| 승인 후 초록색 확정 표시 | 확인 |
| 수작업 바운딩 박스 추가 | 확인 |

## 2단계: 홀(구멍) 개수 — 가공비 산출

1차 목표(OD/ID/폭) 이후, 2단계로 **홀 개수**를 파악하여 가공비를 산출합니다.

<Aside type="note" title="고객 원문 (1차 미팅)">
*"소재 사이즈가 정상적으로 빠지면 그 이후에 구멍들의 개수를 파악해서 구멍당 금액이 어떻게 된다는 둥 접근하는 게 맞겠나 싶다."*
</Aside>

| 항목 | 내용 |
|------|------|
| 홀 판별 기준 | Ø 마크가 있으면 홀 |
| 네모 형태 | 네모도 홀로 표시되는 경우 있음 |
| 중복 표기 | 동일 홀이 여러 번 표기되는 경우 있음 → 개수 파악 필요 |
| 업루트 판단 | *"이거는 좀 확인을 해봐야 한다. 지금은 감이 안 온다."* |
