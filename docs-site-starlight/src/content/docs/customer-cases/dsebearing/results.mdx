### 왜 사람 확인이 필수인가

AI가 치수를 추출하면 사람이 검증 없이 바로 견적에 쓸 수 있을까? **아니다.** 이유는 단순하다.

**90% 정확도의 함정**: 도면 1장에서 80개 치수를 추출하면, 90% 정확도에서도 **8개가 틀릴 수 있다.** 이 8개 중 하나가 외경(Ø704)이면 소재비가 수백만 원 달라진다.

| 오류 종류 | 실제 사례 | 견적 영향 |
|---------|---------|---------|
| 표면거칠기 → 치수 오분류 | `R300`을 직경 300으로 인식 | 소재 사이즈 과대 산정 |
| OCR 숫자 오인식 | `Ø704` → `Ø104` | 소재비 **6배 차이** |
| 공차 기호 무시 | `+0.05/-0.02` 누락 | 가공비 과소 산정 |

**그래서 3단계 검증 구조를 쓴다:**

| 단계 | 처리 | 비중 | 소요 시간 |
|------|------|------|---------|
| L1 자동 승인 | 신뢰도 90% 이상 → 시스템이 바로 통과 | ~70% | 0초 |
| L2 에이전트 검토 | 70~90% → AI가 도면 원본과 대조 | ~20% | 수 초 |
| L3 사람 검토 | 70% 미만 → 담당자가 카드 클릭으로 확인 | ~10% | 치수당 5초 |

**결과**: 80개 치수 기준으로 사람이 직접 봐야 하는 항목은 **8개 내외**. 나머지 72개는 자동 처리된다. 기존 방식(엑셀에 80개 전부 수기 대조)과 비교하면 **검토 시간 90% 절감**이면서, 오류는 잡아낸다.

<Aside type="tip" title="고객에게 설명할 때">
"AI가 100% 맞추는 건 불가능합니다. 대신 **틀릴 가능성이 높은 항목만 골라서 사람에게 보여줍니다.** 전체를 하나하나 확인하는 게 아니라, 의심 항목만 확인하니까 빠르면서도 정확합니다."
</Aside>

## OCR 엔진 교체 용이성

현재 8개 OCR 엔진이 **플러그인 구조**로 연결되어 있다. 특정 모델에 종속되지 않는다.

| 엔진 | 포트 | 용도 | 상태 |
|------|------|------|------|
| **eDOCr2** | 5002 | 기계 도면 치수/GD&T | ✅ 주력 |
| **PaddleOCR** | 5006 | 범용 텍스트 (106 언어) | ✅ 보조 |
| **Ensemble** | 5011 | 4엔진 가중 투표 (최고 정확도) | ✅ 선택 |
| Tesseract | 5008 | 레거시 폴백 | ✅ 대기 |
| TrOCR | 5009 | Transformer 단일행 | ✅ 대기 |
| Surya | 5013 | 문서 레이아웃 | ✅ 대기 |
| DocTR | 5014 | 2단계 파이프라인 | ✅ 대기 |
| EasyOCR | 5015 | CPU 경량 | ✅ 대기 |

**새 OCR 엔진 추가 방법**: YAML 스펙 파일 1개 작성 → 자동 등록. 코드 수정 0줄.

```
api_specs/newocr.yaml → ExecutorRegistry 자동 감지 → GenericAPIExecutor 생성
```

<Aside type="tip" title="고객에게 설명할 때">
