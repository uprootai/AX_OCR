> 원 검출 실패의 근본 원인을 제거한다. OCR에서 Ø를 먼저 찾고, 치수선 방향으로 OD/ID/W를 분류.

![S02 전체 도면 — K원 vs Text-First 타겟](/images/experiments/s02_full.jpg)

전체 도면에서 K가 잡은 케이싱 원(파란)과 S02가 찾는 Ø360 텍스트(녹색 박스). 돋보기로 SECTION 내부를 확대하면 Ø360 텍스트 위치를 확인할 수 있다.

![S02 K vs Text-First 비교](/images/experiments/s02_comparison.jpg)

K method와 S02의 접근법 비교. S02는 원 검출 실패에 면역이고, Ø1036 같은 큰 값도 직접 찾는다.

## 핵심 아이디어

K방법의 약점: **원 검출이 실패하면 전체가 무너진다.**

Text-First는 순서를 뒤집는다:

| 항목 | K (기하학) | S02 (Text-First) |
|------|----------|-------------------|
| 1차 의존 | 원 검출 | **OCR 텍스트** |
| Ø1036 처리 | 스케일 필터 통과 필요 | 바로 OD 후보 |
| 치수선 역할 | 필터링 보조 | **방향 분류 핵심** |
| 원 검출 실패 시 | **전체 실패** | 영향 없음 |

## 파이프라인

```
도면 이미지
  → PaddleOCR / eDOCr2 실행
  → Ø/R 접두사 파싱 (diameter=True / radius=True)
  → 텍스트 bbox 주변 치수선 탐색 (LSD, ±50px 확장)
  → 치수선 방향 분석:
      수평 + Ø → 직경 후보 (OD/ID)
      수직 → W 후보
  → 직경 후보 중 최대=OD, 차대=ID
```

### 분류 기준

| 조건 | 분류 |
|------|------|
| Ø 접두사 + 수평 치수선 | OD/ID 후보 |
| Ø 없어도 수평 + 큰 값 | OD/ID 후보 |
| 수직 치수선 (±15°) | W 후보 |
| 치수선이 원 중심 관통 | confidence +0.3 (보조) |

## 결과

### 87개 배치 기여

| 지표 | K 단독 | K + S02 |
|------|:------:|:-------:|
| OD 검출 | 53/87 (61%) | **73/87 (84%)** |
| 개선 | — | **+20** |

**S02는 K가 OD를 못 찾은 도면에서 가장 효과적이다.** Ø 접두사가 있으면 원 검출 없이 바로 OD 후보로 올린다.

### T5 도면 (GT: OD=1036, ID=580, W=200)

| 값 | S02 결과 | 판정 |
|----|---------|:---:|
| OD | Ø1036 | ✅ |
| ID | 650 | ❌ |
| W | 200 | ✅ |

S01과 동일한 ID 오류. 원 proximity 없이 580 vs 650 구분이 안 된다.

### T1 결과 오버레이

![S02 T1 결과](/images/experiments/s02_result.jpg)

Ø360 텍스트를 직접 발견(녹색 박스). OD=360 ✓, ID=200 ✗, W=190 ✗. 원 검출 없이 OD를 찾았지만 ID/W 분류는 개선 필요.

## 한계

| 한계 | 설명 |
|------|------|
| ID 구분 | S01과 동일 — 원 proximity 필수 |
| Ø 미인식 | OCR이 Ø를 ∅, φ, 0 등으로 오인식하면 놓침 |
| 대각선 치수선 | 수평/수직 외 치수선은 미처리 |
| 참조선 레이아웃 | 텍스트가 치수선 근처에 없는 경우 |

## S01과의 차이

| 항목 | S01 (화살촉) | S02 (Text-First) |
|------|-----------|-------------------|
| 출발점 | 화살촉 검출 | OCR 텍스트 |
| 강점 | W 보완 (+17) | **OD 보완 (+20)** |
| 약점 | ID 구분 | ID 구분 |
| 상호보완 | W 담당 | OD 담당 |

두 방법은 **보완적**이다. 앙상블에서 S01은 W를, S02는 OD를 각각 보완한다.
