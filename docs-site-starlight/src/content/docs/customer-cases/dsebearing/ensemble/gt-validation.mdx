### 동심원 클러스터 — 베어링 자동 식별

메인 뷰에서 원 검출 후 **중심이 같은 원들을 그룹핑** → 가장 큰 동심원 클러스터 = 베어링.

#### 검출 파이프라인 (T1 기준 5단계)

두 가지 원 검출 방법(컨투어+타원피팅, HoughCircles)을 병합 → 중복 제거 → 동심원 클러스터링.

**Step 1 — Canny Edge Detection**

GaussianBlur(5,5) → Canny(50,150) → Dilate. 원/호의 윤곽선을 추출한다.

![Step 1 엣지 검출](/images/gt-validation/steps/step1_edge.jpg)

**Step 2 — Contour + Ellipse Fitting**

컨투어 → `fitEllipse` → 원형도 ≥ 0.70 필터. 타원 장단축 비율이 높은 것만 원으로 인정한다.

![Step 2 컨투어 피팅](/images/gt-validation/steps/step2_contour.jpg)

Step 3부터 두 가지 방안을 비교한다. Step 2의 Contour 결과(2개)는 공통이다.

#### 방안 A — 시각화 개선 (현행 파라미터 유지)

현행 `param2=[60, 100]`을 그대로 쓰되, 신뢰도별 색상을 구분한다.

**Step 3A — HoughCircles (신뢰도별 색상)**

param2=100(빨강, 굵은선) = 높은 신뢰도 19개. param2=60(연한선) = 노이즈 포함 34개. 합계 53개.

![Step 3A 신뢰도별](/images/gt-validation/steps/step3a_hough_viz.jpg)

**Step 4A — Dedup: 55→48개**

중복 제거 후에도 48개로, 노이즈 원이 대부분 살아남는다.

![Step 4A 중복 제거](/images/gt-validation/steps/step4a_dedup_viz.jpg)

**Step 5A — Concentric Clustering**

48개 중 동심원 클러스터 41개. 베어링(초록) 2원(r=114~331)을 잡지만, 나머지 클러스터가 너무 많아 시각적으로 혼잡하다.

![Step 5A 클러스터링](/images/gt-validation/steps/step5a_cluster_viz.jpg)

#### 방안 B — HOUGH_GRADIENT_ALT (동심원 검출)

현행 `HOUGH_GRADIENT`는 2D 누적기(x, y)로 중심을 찾고 중심당 반지름 1개만 선택한다. **동심원은 중심이 같아서 원천적으로 1개만 검출된다.**

`HOUGH_GRADIENT_ALT`는 3D 누적기(x, y, r)를 사용해 같은 중심에서 반지름별로 독립 검출한다.

**Step 3B — HoughCircles ALT (3개)**

기존 53개 → **3개**. 같은 중심(466, 330)에서 r=245, r=197, r=102 — 동심원 3개를 정확히 잡는다.

![Step 3B ALT](/images/gt-validation/steps/step3b_hough_tuned.jpg)

**Step 4B — Dedup: 5→5개**

Contour 2개 + Hough ALT 3개 = 5개. 중복 없이 전부 유효.

![Step 4B 중복 제거](/images/gt-validation/steps/step4b_dedup_tuned.jpg)

**Step 5B — Concentric Clustering**

5개 → 클러스터 2개. 베어링(초록) **4원**(r=102~285), ID/OD 비율 0.358 (GT=0.528).

![Step 5B 클러스터링](/images/gt-validation/steps/step5b_cluster_tuned.jpg)

#### A/B 비교 요약

| 항목 | 방안 A (현행 GRADIENT) | 방안 B (GRADIENT_ALT) |
|------|:---:|:---:|
| 알고리즘 | HOUGH_GRADIENT (2D 누적기) | HOUGH_GRADIENT_ALT (3D 누적기) |
| Hough 파라미터 | param2=[60,100], dp=1.5 | param2=0.85, dp=1.5 |
| Hough 검출 원 | 53개 | **3개** (동심원) |
| Dedup 후 | 48개 | 5개 |
| 클러스터 수 | 41개 | 2개 |
| 베어링 원 | 2개 (r=114~331) | **4개** (r=102~285) |
| ID/OD 비율 | 0.345 | 0.358 |
| GT 비율 | 0.528 | 0.528 |
| 시각 품질 | 혼잡 | 깔끔 |
| 동심원 검출 | 불가 (중심당 1원) | **가능** (중심당 반지름별) |

방안 B(`HOUGH_GRADIENT_ALT`)가 노이즈 없이 동심원을 직접 검출한다. 현행 방안 A의 근본 한계 — **중심이 같은 원을 못 잡는 문제** — 를 해결한다.

### Cardinal Projection v3 — ALT 동심원 기반 화살촉 매칭

S08 실험에서 Cardinal Projection은 **원 검출 부정확**으로 실패했다 (v1: 774개 과다 매칭, v2: 0개 매칭). ALT 동심원으로 원 검출 문제가 해결됐으므로 v3를 재시도한다.

#### 알고리즘

```
1. HOUGH_GRADIENT_ALT → 동심원 3개 (r=외곽, 중간, 내곽)
   ↓
2. 각 원의 N/S/E/W 끝점 (center ± r)
   ↓
3. 각 끝점에서 원 바깥쪽으로만 직선 배치
   N/S → 수평선 (좌우 바깥), E/W → 수직선 (상하 바깥)
   ↓
4. 직선 15px 이내의 화살촉 후보 검색 (Black Hat 모폴로지)
   ↓
5. 가장 가까운 화살촉 = 히트
```

#### v1/v2 → v3 비교

| 항목 | v1 (S08) | v2 (S08) | **v3 (ALT)** |
|------|:---:|:---:|:---:|
| 원 검출 | Contour+Hough 20개 | 동심 필터 3개 | **ALT 3개** |
| 원 정확도 | 케이싱 잡음 | 케이싱 잡음 | **베어링 동심원** |
| 매칭 | 774개 (과다) | 0개 (과소) | **42/48 (88%)** |

#### 시각화 범례

| 요소 | 의미 |
|------|------|
| 초록 원 | ALT 동심원 (베어링) |

| 노랑 선 | E/W 끝점의 **수직선** (원 바깥쪽만) |

| 빨간 X | 화살촉 히트 |

#### 분석

v3에서 히트율 88%를 달성한 핵심 요인:
1. **ALT 동심원**: 원 끝점 좌표가 정확
2. **원 바깥쪽 직선**: N/S→수평선, E/W→수직선을 원에 안 닿게 바깥으로만 배치
3. **원 내부 필터**: 원 반지름 90% 이내 화살촉 제외 → 노이즈 감소
4. **적정 tolerance(15px)**: 직선과 화살촉 간 수직 거리만 체크

미히트 원인 (6/48):
- N 방향 외원(r=245): 상단 보조도 영역에 화살촉 부재
- 일부 히트(d>80px)는 노이즈 가능성 — 향후 거리 기반 필터링 필요
- **치수선 추적 → OCR 매칭** 단계 추가 시 실제 치수값 추출 가능

#### 전체 파이프라인 — 동심원 → 투사 → 선분 추적 → OCR

투사선 위 화살촉에서 Line Detector(5016)로 연결된 선분을 찾고, 선분 반대쪽 끝의 화살촉을 쌍으로 매칭한 뒤, 치수선 중점 근처 OCR 텍스트를 추출한다.

```
[1] ALT 동심원 → [2] 전체 도면 화살촉 → [3] 투사선 히트(끝점당 1개)
→ [4] Line Detector 선분 → 쌍 화살촉 → [5] OCR 매칭
```

#### 현재 한계 — OD/ID 미검출 원인

T1 전체 도면 OCR 결과에서 OD=360, ID=190은 **치수선으로 표기되지 않는다**:

| OCR 텍스트 | 위치 | 비고 |
|-----------|------|------|
| `T1 BEARING ASSY (360X190)` | 타이틀 블록 | OD/ID가 세션명에만 존재 |
| `(250)` | SECTION 우측 | 참조 치수 |
| `(200)` | SECTION 상/하 | **W 후보 ✓** |
| `M48` | 메인 뷰 | 볼트 규격 |

ASSY 도면에서 OD/ID는 치수선+화살촉이 아닌 **세션명 텍스트**로만 표기되는 구조. Cardinal 투사 → 화살촉 쌍 방식으로는 W만 추출 가능하고, OD/ID는 세션명 파싱 또는 SECTION 내 Ø 치수 검출이 필요하다.

:::tip[S08 → v3 전환의 의미]
S08의 "원이 틀리면 전체가 틀린다"는 근본 한계가 `HOUGH_GRADIENT_ALT`로 해결됐다. 동심원 검출 → Cardinal 투사 → 화살촉 매칭 → OCR 치수 추출 파이프라인이 현실적으로 가능해졌다. 다만 ASSY 도면의 OD/ID는 치수선이 아닌 세션명에 있어, 기존 앙상블 K 방법과 병행이 필요하다.
:::

## GT-2: TD0062055 — Radial Bearing

Dimension Lab에서 TD0062055를 선택하면, GT가 설정된 상태로 표시된다: **OD(외경)=515, ID(내경)=440, W(폭)=48**.

![GT-2 도면 선택 — GT 설정 완료 (OD=515, ID=440, W=48)](/img/ensemble/gt2-with-gt-set.png)

### 7×14 매트릭스 결과

K.기하학기반이 **전 엔진 100%** 정확도를 달성한다. OD/ID는 다수 방법에서 100% 일치하나, W=48을 정확히 검출하는 것은 K 방법 뿐이다.

![GT-2 매트릭스 — K.기하학기반 전 엔진 100%, OD/ID 다수 방법 100%](/img/ensemble/gt2-matrix-with-gt.png)

### 버전별 결과

| 버전 | OD | ID | W | 정확 |
|------|:---:|:---:|:---:|:---:|
| v2 (투표) | Ø515 ✅ | 440 ✅ | **250** ❌ | 2/3 |
| v3 (K-priority) | Ø515 ✅ | 440 ✅ | **250** ❌ | 2/3 |
| v5 (최적화) | Ø515 ✅ | 440 ✅ | **48** ✅ | **3/3** |

**v5에서 W가 수정된 핵심 사례.**

### W 선택 과정 비교

**v3 (잘못된 결과)**:
- `ID/OD = 440/515 = 0.854 > 0.75` → 스러스트 경로
- 스러스트: 최대 W 선택 → **W=250**

**v5 (올바른 결과)**:
- `expected_w = (515 - 440) / 2 = 37.5`
- W 후보: 250, 90, 48.5, ...
- 48.5: `|48.5 - 37.5| / 37.5 = 0.29` → 50% 범위 내 ✅
- 250: `|250 - 37.5| / 37.5 = 5.67` → 범위 밖 ❌
- radial 공식으로 **W=48** 선택 (48.5 반올림)

### 방법별 결과

| 방법 | OD | ID | W |
|------|:---:|:---:|:---:|
| K | Ø515 ✅ | 440 ✅ | 48 ✅ (v5) |
| S01 | Ø515 ✅ | 440 ✅ | — |
| S02 | Ø515 ✅ | 440 ✅ | — |

S01/S02는 W를 검출하지 못했다. K만이 W를 제공하므로 K-priority의 W 분류 정확도가 핵심.

## GT 확보의 한계

### BOM size 필드 ≠ GT

초기에 BOM `bom_items.json`의 `size` 필드에서 22개 GT 후보를 추출했다.

예시:
| 도면 | BOM size | 앙상블 OD | 일치 |
|------|---------|-----------|:---:|
| TD0062039 | 660×350×130 | Ø635 | ❌ |
| TD0062046 | 290×210×30 | Ø260 | ❌ |

분석 결과, BOM size는 **소재 규격**(가공 전 원자재 치수)으로 확인됐다.
완제품 치수와 수 mm~수십 mm 차이가 나므로 직접 GT로 사용할 수 없다.

### GT 확보 방법

| 방법 | 난이도 | 신뢰도 | 현실성 |
|------|:---:|:---:|:---:|
| ASSY 도면 설명 텍스트 | 낮음 | 높음 | ~7개 (현재 2개 확보) |
| 고객 제공 치수표 | 낮음 | 최고 | 고객 협조 필요 |
| 수동 측정 (도면 눈대중) | 중간 | 중간 | 87개 전수 가능하나 노동집약 |
| BOM size 범위 검증 | 낮음 | 낮음 | ±10% 범위 체크용으로만 |
