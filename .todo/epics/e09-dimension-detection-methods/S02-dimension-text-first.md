# S02: Dimension-Text-First — OCR 먼저, 치수선 역추적

> 원 검출 의존성을 완전 제거. OCR 결과에서 Ø 접두사 값을 먼저 찾고, 텍스트 bbox 주변의 치수선을 역추적하여 OD/ID/W를 분류한다.

## 왜 이 방법인가

현재 파이프라인: 원 검출 → 스케일 추정 → OCR → 분류 (원 검출이 기초)
제안 파이프라인: **OCR → Ø/R 텍스트 식별 → 치수선 추적 → 방향 분석 → 분류** (원 검출 불필요)

장점:
- 원 검출 오류가 전파되지 않음 (T5 패턴 근본 해결)
- Ø1036이 OCR에서 검출되면 바로 OD 후보
- 치수선 방향으로 R(반지름) vs D(직경) 직접 판단

## 핵심 참고 자료

- eDOCr2 텍스트 검출 파이프라인 (이미 사용 중)
- M방법(심볼검색)의 확장: Ø 검출 후 "어디에 연결되는지"까지 추적
- [Dimension-Text-First approach in arrowhead literature](https://www.sciencedirect.com/science/article/abs/pii/S0167865597000251)

## 구현 계획

### Step 1: 전체 OCR + Ø/R 파싱 (1h)

```
입력: 도면 이미지
  → PaddleOCR / eDOCr2 실행
  → 각 텍스트에서 Ø/R 접두사, 숫자값, 공차 파싱
  → Ø 있으면 diameter=True, R 있으면 radius=True
출력: List[{text, num_val, is_diameter, is_radius, bbox, confidence}]
```

### Step 2: 텍스트 bbox 주변 치수선 탐색 (2h)

```
입력: OCR 결과 + LSD 검출 선분
  → 각 텍스트 bbox를 확장(±50px)
  → 확장 영역과 교차하는 선분 찾기
  → 교차 선분 중 가장 긴 것 = 해당 텍스트의 치수선
출력: List[{text, dim_line, line_direction, line_length}]
```

### Step 3: 치수선 방향 분석 → OD/ID/W 분류 (2h)

```
입력: 텍스트-치수선 매칭 결과
  → 수평 치수선 (±15°) + Ø 접두사 → 직경 후보 (OD/ID)
  → 수직 치수선 (±15°) → W 후보
  → 치수선 양 끝이 원주 근처 → OD/ID 강화
  → Ø 접두사 없어도 수평 + 큰 값 → OD/ID 후보
  → 직경 후보 중 최대=OD, 차대=ID
출력: {od, id, w}
```

### Step 4: 원 검출 보조 (선택적) (1h)

```
원 검출 결과가 있으면 보조 근거로 활용:
  → 치수선이 원 중심을 관통하면 confidence +0.3
  → 관통하지 않아도 분류 결과는 유지 (원 검출 의존 아님)
```

### Step 5: 통합 + 배치 테스트 (1h)

- 새 방법 `P` (text-first)로 등록
- 87개 도면 배치 평가
- K방법 및 기존 M방법과 비교

## 예상 소요: 7시간

## K방법과의 차이점

| 항목 | K (기하학기반) | P (Text-First) |
|------|-------------|-----------------|
| 1차 의존 | 원 검출 | OCR 텍스트 |
| Ø1036 처리 | 스케일 필터 통과 필요 | 바로 OD 후보 |
| 치수선 역할 | 필터링 보조 | 방향 분류 핵심 |
| 원 검출 실패 시 | 전체 실패 | 영향 없음 |

## 리스크

- OCR이 Ø를 인식하지 못하면 M방법과 동일한 한계
- 치수선이 텍스트 근처에 없는 레이아웃(참조선 사용)
- 수평/수직 외 대각선 치수선 처리

## 성공 기준

- Ø 접두사 검출률 > 90% (OCR 기반)
- 텍스트-치수선 매칭률 > 60%
- OD 정확도에서 K방법과 동등 또는 향상
